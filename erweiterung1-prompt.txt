Wichtige Architektur-Entscheidung – bitte jetzt anpassen:

Wir gehen bewusst den hybriden Weg (Option 1):

**Kern-Architektur:**
- Die Extension ist **kein** vollständiger Nachbau der Agent-Logik.
- Stattdessen nutzen wir den **installierten Grok Build CLI** als Backend über das **Agent Client Protocol (ACP)** / `grok agent stdio` (JSON-RPC).
- Die Extension übernimmt die **native VS Code Experience** (UI, Context-Handling, Multi-Session-Management, Diff-Views, Integration in den Editor).
- Der eigentliche Agent (Planung, Tool-Ausführung, Reasoning, Safety etc.) bleibt beim offiziellen Rust-Code von Grok Build.

Das hat folgende Vorteile:
- Wir bekommen exakt das gleiche Verhalten wie das originale Grok Build.
- Wir müssen die komplexe Agent-Logik nicht selbst nachbauen.
- Die Extension wird trotzdem sehr "native" und angenehm in VS Code anfühlen.

**Bitte passe deine bisherige Planung jetzt entsprechend an:**

1. **Architektur anpassen**
   - Die Extension kommuniziert primär mit dem `grok` Binary über ACP.
   - Baue ein sauberes Session-Management, das mehrere gleichzeitige `grok agent` Prozesse starten und steuern kann.
   - Jede Session läuft in einem eigenen Webview-Panel (Sidebar + optional separate Editor Group oder Auxiliary Window).

2. **Graceful Handling des CLI**
   - Beim ersten Start prüfen, ob `grok` installiert ist.
   - Falls nicht: Automatisch Installationsanleitung anbieten (oder sogar den Install-Befehl ausführen, wo möglich).
   - Guter Fallback / klare Fehlermeldungen, falls der CLI nicht gefunden wird.

3. **VS Code Native Stärken maximal ausnutzen**
